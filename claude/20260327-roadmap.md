# Claude Code Usage Tracker -- Feature Roadmap

This roadmap covers 16 features derived from 12 concerns and 17 opportunities identified in the 2026-03-27 codebase investigation. The project is a single-user, local-only web app (~4,800 LoC TypeScript, 3 runtime deps) that parses Claude Code JSONL session files and presents a cost/usage dashboard. Features are ordered by difficulty (easiest first), with dependency and impact used as tiebreakers. Every concern and opportunity from the investigation maps to at least one feature.

## Difficulty Scale

| Rating | Meaning |
|--------|---------|
| 1/5 | Trivial -- a few hours, minimal risk |
| 2/5 | Easy -- a focused day or two |
| 3/5 | Moderate -- multi-day, some coordination between layers/systems |
| 4/5 | Hard -- significant effort, architectural decisions required |
| 5/5 | Very Hard -- major undertaking, weeks of work |

Half-steps (1.5, 2.5, etc.) are used when a feature sits between two levels.

## 2. Localhost Binding and Shutdown Auth

**Difficulty: 1/5**

### Overview

Bind the Express server to `127.0.0.1` instead of `0.0.0.0` so it is only accessible from the local machine. Optionally add a shared-secret check to the `POST /api/shutdown` endpoint. The primary fix is the host binding (one line); the shutdown auth is belt-and-suspenders for a local tool.

### Pros

- Eliminates the LAN exposure vector entirely -- no device on the network can reach the API
- Single-line change to `app.listen()` for the host binding
- The shutdown endpoint becomes inaccessible from outside the machine regardless of auth
- Appropriate security posture for a personal tool that handles no secrets

### Cons

- If the user ever wants to access the dashboard from another device on their network (e.g., phone, tablet), this blocks it. There is no override mechanism unless they change the bind address back or add a CLI flag
- The localhost binding means the `sync-on-exit.sh` script must use `localhost` or `127.0.0.1` in its curl calls, not a hostname that might resolve differently -- worth verifying

### What This Touches

- `src/server/index.ts:47` -- add `'127.0.0.1'` as second argument to `app.listen(PORT, '127.0.0.1', ...)`
- `scripts/sync-on-exit.sh` -- verify all curl calls target `localhost` (they likely already do)
- Optionally: `src/server/index.ts:31-37` -- add a token check to the shutdown handler, with the token generated on server start and written to a known file path

### Impact Analysis

| Metric | Before | After | Delta |
|--------|--------|-------|-------|
| Network-accessible endpoints | 17 (all, from LAN) | 17 (localhost only) | LAN exposure eliminated |
| Shutdown endpoint auth | None | Localhost-restricted (+ optional token) | Risk reduced from "any LAN device" to "local processes only" |

No user-visible UI impact. The app continues to work identically at `http://localhost:3000`.

### Broad Todo List

**Server:**
- Add `'127.0.0.1'` host parameter to `app.listen()` in `src/server/index.ts`
- Verify the server startup log message still shows the correct URL

**Scripts:**
- Review `scripts/sync-on-exit.sh` to confirm all HTTP calls use `localhost` or `127.0.0.1`

**Testing:**
- Start the server and confirm it is reachable from `localhost:3000`
- Confirm it is NOT reachable from another device on the LAN (or from the machine's LAN IP)

### Additional Notes

Covers: SEC-01 (C-02). The host binding alone is sufficient for a local tool. Adding token-based auth to `/api/shutdown` is optional -- localhost restriction already makes it accessible only to local processes. If the user later wants remote access, a `--host 0.0.0.0` CLI flag would be the right escape hatch.

- If the project later decides to materialize daily stats for performance (the original design intent), the table would need to be re-created. This is trivial but worth noting

### What This Touches

- `src/server/db/schema.ts:87-96` -- remove the CREATE TABLE statement for `daily_stats`
- `src/server/db/schema.ts` (migration section) -- add a `DROP TABLE IF EXISTS daily_stats` statement

### Impact Analysis

| Metric | Before | After | Delta |
|--------|--------|-------|-------|
| DB tables | 7 | 6 | -1 |
| Schema LoC | ~100 | ~90 | ~-10 |

No user-visible impact. The table is empty and unused.

### Broad Todo List

**Schema:**
- Remove the `daily_stats` CREATE TABLE statement from `src/server/db/schema.ts`
- Add `DROP TABLE IF EXISTS daily_stats` to the migration section of schema initialization
- Verify no other file references `daily_stats` (grep for it)

**Testing:**

### Additional Notes

## 6. Cost SQL Deduplication

**Difficulty: 2/5**

### Overview

Replace the two hand-maintained SQL CASE expression strings (`MESSAGE_COST_SQL` and `MESSAGE_COST_NO_CACHE_SQL`) with a single pricing data structure and a generator function that produces both SQL variants. Currently, adding a new model requires updating two parallel 15-line SQL strings in lockstep with different cache rate multipliers -- error-prone and tedious.

### Pros

- Adding a new model tier becomes a one-line change to the pricing data structure instead of editing two SQL strings
- Eliminates the risk of the two SQL variants drifting out of sync (e.g., adding a model to one but forgetting the other)
- Makes the pricing logic inspectable as data rather than embedded in SQL strings
- Enables downstream features (Feature #10, Dynamic Model Pricing Registry) to build on this structure

### Cons

- The generated SQL must produce identical output to the hand-written SQL. Debugging a SQL generation function is harder than debugging a SQL string -- if the output is wrong, you need to reason about the generator, not just the SQL
- Adds a layer of indirection: developers reading the code see a function call rather than the raw SQL, which may be less immediately understandable
- The current two-string approach is ugly but works correctly and is well-tested (36 query tests cover pricing). Refactoring working, tested code has non-zero risk

### What This Touches

- `src/server/db/queries.ts:5-47` -- replace `MESSAGE_COST_SQL` and `MESSAGE_COST_NO_CACHE_SQL` with a pricing data structure and a `buildCostSql(useCache: boolean)` function
- All queries that reference these SQL strings remain unchanged (they just reference the function output instead of the raw string)
- `src/server/db/queries.test.ts` -- existing pricing tests serve as the regression suite; no new tests needed but consider adding a direct test for the generator function

### Impact Analysis

| Metric | Before | After | Delta |
|--------|--------|-------|-------|
| Lines of pricing SQL | ~35 (two parallel 15-line CASE expressions) | ~20 (data structure) + ~15 (generator) = ~35 | Neutral LoC, but single source of truth |
| Cost to add new model | Edit 2 SQL strings in lockstep | Add 1 entry to pricing array | Reduced from error-prone to trivial |

No user-visible impact. Internal code quality improvement.

### Broad Todo List

**Data Structure:**
- Define a `ModelPricing` interface: `{ pattern: string, inputRate: number, outputRate: number, cacheWriteMultiplier: number, cacheReadMultiplier: number, contextThreshold?: number }`
- Create a `MODEL_PRICING` array with the 5 current model tiers (Sonnet 4.5 high-context, Sonnet 4.5, Opus 4.5, Haiku 4.5, Sonnet 4) plus the ELSE default
- Ensure CASE ordering is preserved (more specific patterns before broader ones)

**Generator:**
- Implement `buildCostSql(useCache: boolean): string` that iterates the pricing array and produces a SQL CASE expression
- When `useCache` is true, apply cache write multiplier (1.25x) and cache read multiplier (0.1x)
- When `useCache` is false, use the base input rate for all token types

**Migration:**
- Replace the two raw SQL string constants with calls to `buildCostSql(true)` and `buildCostSql(false)`
- Run the full test suite to confirm all 36 pricing tests pass with identical results

### Additional Notes

Covers: ARCH-02 (C-05). This is a prerequisite for Feature #10 (Dynamic Model Pricing Registry). The generator function makes it possible to later load pricing from the database or a config file rather than hardcoding it.

## 8. Incremental Sync File Offset Optimization

**Difficulty: 2.5/5**

### Overview

Fix the incremental sync to actually use the stored byte offset when re-reading session files. Currently, `parseSessionFile()` reads the entire file with `readFileSync` regardless of the stored `startOffset`, then parses all lines from the beginning. The offset is only used for an early-return check and for updating `sync_state`. The fix: either use `fs.createReadStream` with a `start` byte offset, or read the file and slice the buffer from the offset before splitting into lines.

### Pros

- For re-synced files, avoids re-reading and re-parsing content that was already processed. A 5MB session file that grew by 100KB would only parse the new 100KB instead of the full 5MB
- Reduces CPU time proportionally to how much of the file is already processed
- The upsert-based deduplication already prevents duplicate DB writes, so this optimization is purely about I/O and parsing savings
- Particularly valuable for the `sync-on-exit.sh` hook, which syncs the just-completed session on every Claude Code exit

### Cons

- Splitting a JSONL file at an arbitrary byte offset risks landing in the middle of a line. The implementation must handle partial lines at the offset boundary (discard the first incomplete line, or seek backward to the previous newline)
- The streaming deduplication logic (last occurrence of a message ID wins) depends on seeing all messages in a session. If we only parse from the offset, we lose the ability to overwrite earlier messages with later ones. This may not matter if deduplication is only relevant within a single sync batch, but it needs verification
- `readFileSync` is currently simple and reliable. Switching to offset-aware reading adds an edge case (the boundary handling) that needs test coverage

### What This Touches

- `src/server/parser/jsonl.ts:130-145` (approximate) -- the file reading and line splitting section
- `src/server/parser/jsonl.ts` -- the `startOffset` parameter usage throughout the function
- `src/server/parser/jsonl.test.ts` -- new tests for offset-based parsing, partial line handling, and boundary conditions

### Impact Analysis

| Metric | Before | After | Delta |
|--------|--------|-------|-------|
| File I/O per incremental sync | Read entire file | Read from offset to EOF | Proportional to new content only |
| Parse work per incremental sync | All lines in file | Only lines after offset | Same proportion |
| DB write behavior | Unchanged (upserts) | Unchanged | No change |
| Test count | 66 | ~70-72 | +4 to +6 for offset edge cases |

User-visible impact: faster sync for large session files. For typical small files (sub-MB), the difference is imperceptible.

### Broad Todo List

**Parser:**
- Modify the file reading in `parseSessionFile()` to use a buffer slice from `startOffset` instead of reading from byte 0
- Handle the partial-line boundary: if `startOffset > 0`, discard bytes before the first newline after the offset
- Verify that the streaming message deduplication (Map keyed by message ID) still works correctly when only parsing from an offset

**Testing:**
- Add a test for incremental parse from a mid-file offset
- Add a test for the partial-line boundary case (offset lands in the middle of a JSON line)
- Add a test for offset at exactly a line boundary (no partial line)
- Add a test where the offset equals the file size (no new content -- should be a no-op)

**Validation:**
- Run the full test suite
- Manually test with a real session file: sync once, append content, sync again, verify only new content is parsed (add debug logging if needed)

### Additional Notes

Covers: PERF-01 (C-03). The simplest approach is `const buf = fs.readFileSync(filePath); const content = buf.slice(startOffset).toString('utf-8');` followed by discarding the first partial line. This avoids the complexity of `createReadStream` while still eliminating re-parsing. The `createReadStream` approach with `{ start: offset }` is more memory-efficient for very large files but adds async complexity to a currently synchronous codebase.

## 9. Server-Side Pagination

**Difficulty: 2.5/5**

### Overview

Add `LIMIT` and `OFFSET` SQL clauses to `getSessionStats()` and `getDailyStats()` so the server returns only one page of results at a time. Currently, both functions return the full dataset and the client paginates by slicing the array. This is fine for hundreds of sessions but degrades for thousands.

### Pros

- Reduces JSON response size proportionally -- instead of returning all sessions, returns only the requested page (25-100 rows)
- Enables the application to handle thousands of sessions without the client loading all data into memory
- The database already has indexes on the relevant columns, so LIMIT/OFFSET queries will be fast

### Cons

- Requires API contract changes: the endpoints need to accept `page` and `pageSize` query parameters and return a `total` count alongside the data. This is a breaking change to the API response format
- The client-side pagination logic in `SessionList.tsx` and `DailyStatsTable.tsx` must be replaced with server-driven pagination (fetch on page change, not slice). This changes the data flow significantly
- Two concurrent requests may be needed: one for the page data and one for the total count (or one query with a window function for total). The window function approach adds complexity to the already-heavy `getSessionStats` query
- Filters that currently compose client-side (date range, project, custom title) now need to be passed as query parameters and applied in the SQL WHERE clause on every paginated request

### What This Touches

- `src/server/db/queries.ts` (or `stats-queries.ts` after Feature #7) -- modify `getSessionStats()` and `getDailyStats()` to accept `limit` and `offset` parameters, add `COUNT(*)` for total
- `src/server/routes/stats.ts` -- parse `page` and `pageSize` query parameters, pass to query functions, return `{ data, total, page, pageSize }` response shape
- `src/client/components/SessionList.tsx` -- replace client-side pagination with server-driven: fetch on page change, display total from server response
- `src/client/components/DailyStatsTable.tsx` -- same server-driven pagination changes
- `src/client/components/Dashboard.tsx` -- adjust data fetching to pass pagination params, update state management for paginated data

### Impact Analysis

## 16. Migrate to Drizzle ORM

**Difficulty: 3.5/5**

### Overview

Replace all raw `better-sqlite3` query calls with Drizzle ORM. This involves defining the existing 7 tables as Drizzle schema (in TypeScript), replacing raw SQL strings with Drizzle's type-safe query builder, and swapping `better-sqlite3` for Drizzle's `drizzle-orm/better-sqlite3` driver (or `drizzle-orm/node-postgres` if switching to Postgres). Drizzle Kit provides migration tooling to manage schema changes going forward.

### Pros

- Type-safe queries -- eliminates runtime SQL errors and gives autocomplete on column names, joins, and return types
- Schema defined in TypeScript -- single source of truth for table structure, shared across server code
- Migration tooling via Drizzle Kit -- generates SQL migration files from schema diffs, no more manual `CREATE TABLE` statements
- Lightweight -- ~50KB, no binary dependencies, no code generation step (unlike Prisma)
- Keeps the SQL mental model -- Drizzle queries read like SQL, so the existing query logic translates directly
- Opens the door to Postgres migration later -- Drizzle supports both SQLite and Postgres with the same schema API, making a future database switch much simpler
- Zero runtime overhead compared to heavier ORMs

### Cons

- Every query in `queries.ts` (or post-split modules from Feature #7) must be rewritten. There are ~20-25 query functions, each with raw SQL
- The cost SQL (deduplicated in Feature #6) is complex with model-specific pricing CASE expressions -- translating this to Drizzle's query builder may be less readable than raw SQL
- Learning curve for the team (though minimal given Drizzle's SQL-like API)
- Adds 1 runtime dependency (`drizzle-orm`) and 1 dev dependency (`drizzle-kit`)
- Existing tests that assert on raw SQL behavior will need updating
- Risk of subtle behavioral differences in query output (e.g., NULL handling, type coercion) during migration

### What This Touches
